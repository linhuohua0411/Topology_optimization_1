# Hyperledger Fabric Virtual Network

> **广域网仿真环境下的 Hyperledger Fabric 2.5 虚拟化区块链网络**

本项目是国家重点研发计划公共平台的组成部分，在 Docker 容器化环境中构建基于 BIRD/BGP 的多自治系统（AS）广域网仿真拓扑，并在其上部署完整的 Hyperledger Fabric 2.5 区块链网络。项目提供两档规模：小规模开发拓扑（2 AS / 6 节点）和大规模生产拓扑（12 AS / 108 节点），支持可控的网络延迟、带宽、丢包实验，适用于联盟链性能评估、共识协议分析与跨域区块链网络研究。

## 1. 项目概述

### 1.1 背景与目标

在区块链网络研究中，真实广域网环境部署成本高、可重复性差、参数不可控。本项目通过 **Docker 容器 + BIRD BGP 路由协议**，在单台服务器上构建逼真的多 AS 广域网拓扑，将 Hyperledger Fabric 节点（CA、Orderer、Peer）部署到不同的虚拟自治系统中，实现：

- **可控的网络条件**：精确调节跨 AS 延迟（1ms–300ms）、带宽（5Mbit–1Gbit）、丢包率（0%–5%）
- **可复现的实验环境**：所有配置脚本化，支持一键部署与销毁
- **规模化部署**：单服务器可运行 148 个容器（103 Peer + 5 Orderer + 12 CA + 24 路由器 + 监控栈）
- **真实的路由行为**：BGP 路由收敛、OSPF 内部路由、AS 间路径选择等网络行为与真实互联网一致

### 1.2 核心创新


| 特性             | 说明                                                   |
| -------------- | ---------------------------------------------------- |
| BIRD/BGP 广域网仿真 | 多层 AS 拓扑（核心/转接/客户），真实 BGP 路由收敛                       |
| Fabric-on-WAN  | Orderer Raft 集群跨 AS 部署，Gossip 协议经 BGP 路由传播           |
| 参数化网络控制        | 6 种预设 + 自定义延迟/带宽/丢包，支持实验对照组设计                        |
| 全栈监控           | Prometheus + Grafana + 自研 Fabric Monitor，覆盖共识/背书/网络层 |
| 双规模拓扑          | 小规模（2 AS / 6 节点）快速迭代，大规模（12 AS / 108 节点）生产验证         |

## 3. 与真实 Hyperledger Fabric 部署的对比

### 3.1 架构差异


| 维度       | 真实生产部署                     | 本仿真方案                           | 差异说明                                                           |
| -------- | -------------------------- | ------------------------------- | -------------------------------------------------------------- |
| **网络层**  | 物理 WAN / VPN / 专线          | Docker 网络 + BIRD BGP + tc qdisc | 仿真方案通过 BGP 协议栈模拟真实 AS 路径选择，tc netem 注入延迟/丢包，路由行为真实但物理层为共享宿主机网卡 |
| **部署方式** | 物理机 / Kubernetes / 云服务     | Docker Compose 单宿主机             | 仿真方案将所有节点集中在一台服务器，通过容器网络隔离模拟地理分布                               |
| **存储**   | 独立物理磁盘 / SAN / NFS         | Docker Volume (共享宿主机磁盘)         | I/O 竞争在大规模下可能影响性能指标，但不影响功能正确性                                  |
| **安全边界** | 防火墙 / VPN / mTLS / HSM     | Docker 网络隔离 + TLS               | 容器间通信经过 Docker 桥接网络，TLS 加密与真实环境一致                              |
| **服务发现** | DNS / Consul / K8s Service | Docker extra_hosts + 静态 IP      | 使用静态 IP 分配，通过 extra_hosts 配置主机名解析                              |
| **证书管理** | Fabric CA 动态注册 + 自动续期      | cryptogen 静态生成                  | 仿真方案使用 cryptogen 一次性生成所有证书，适合实验但不适合长期运行                        |
| **运维**   | 日志聚合 / 告警 / 自动扩缩容          | Prometheus + Grafana + 脚本监控     | 提供基础监控能力，无自动恢复机制                                               |


### 3.2 性能差异


| 指标           | 真实 WAN 环境       | 本仿真方案                  | 影响评估                                              |
| ------------ | --------------- | ---------------------- | ------------------------------------------------- |
| **网络延迟精度**   | 真实延迟含抖动、拥塞、路由变化 | tc netem 固定延迟 + 可选抖动   | 仿真延迟更稳定可控，适合对照实验；真实 WAN 的长尾延迟无法完美模拟               |
| **带宽隔离**     | 物理链路独享带宽        | tc tbf 限速，宿主机网卡共享      | 多容器并发时可能出现带宽竞争，但对 Fabric 交易流量（KB 级）影响可忽略          |
| **CPU/内存隔离** | 独立物理资源          | 容器共享宿主机资源              | 103 Peer 同时运行时 CPU 调度引入微秒级抖动；建议测试机 32 核 / 64GB 以上 |
| **存储 I/O**   | 独立 SSD，IOPS 确定  | 共享磁盘，Docker 存储驱动开销     | LevelDB/CouchDB 写入密集时可能出现 I/O 等待，建议使用 SSD         |
| **吞吐量上限**    | 受限于网络带宽和节点处理能力  | 受限于宿主机 CPU 和 Docker 调度 | 单机 TPS 上限低于分布式部署，但足以验证协议行为                        |


**适用场景判断：**


| 研究目的          | 是否适用 | 说明                      |
| ------------- | ---- | ----------------------- |
| 共识协议行为分析      | 适用   | Raft 选举、日志复制行为与真实环境一致   |
| 网络延迟对 TPS 的影响 | 适用   | tc netem 精确控制延迟，可设计对照实验 |
| 大规模网络拓扑验证     | 适用   | 12 AS / 103 Peer 拓扑规模足够 |
| 绝对性能基准测试      | 不适用  | 单机资源共享导致绝对值偏低，应关注相对趋势   |
| 生产环境容量规划      | 不适用  | 仿真环境的资源竞争模式与生产环境不同      |
| 安全攻防实验        | 部分适用 | 网络隔离为逻辑隔离，非物理隔离         |


### 3.3 功能完整性对比


| Fabric 功能          | 真实部署                 | 本仿真方案                | 状态        |
| ------------------ | -------------------- | -------------------- | --------- |
| **Raft 共识**        | 多节点 Raft             | 3/5 节点 Raft 跨 AS     | 完整实现      |
| **通道管理**           | 多通道                  | 单通道 (fabricchannel)  | 简化实现      |
| **链码生命周期**         | 完整 2.x 生命周期          | 打包→安装→审批→提交          | 完整实现      |
| **TLS 通信**         | mTLS 端到端             | TLS 加密（cryptogen 证书） | 完整实现      |
| **Gossip 协议**      | 自动发现 + 选举            | Leader 选举 + 锚节点      | 完整实现      |
| **背书策略**           | 任意策略表达式              | OR / AND 策略          | 完整实现      |
| **MSP 身份管理**       | Fabric CA 动态注册       | cryptogen 静态生成       | 简化实现      |
| **私有数据集合**         | 完整支持                 | 未配置                  | 未实现（可扩展）  |
| **CouchDB 状态库**    | 可选 CouchDB           | 默认 LevelDB           | 简化实现（可切换） |
| **链码即服务 (CaaS)**   | K8s 外部构建器            | Docker-in-Docker 构建  | 简化实现      |
| **Fabric Gateway** | SDK Gateway 模式       | CLI 直接调用             | 简化实现      |
| **HSM 硬件安全**       | PKCS#11 集成           | 软件密钥存储               | 未实现       |
| **操作控制台**          | Hyperledger Explorer | Grafana + 自研 Monitor | 替代实现      |


### 3.4 研究价值说明

尽管仿真方案在物理资源隔离和绝对性能上与真实部署存在差异，但其核心价值在于：

1. **协议层面完全一致**：Raft 共识、Gossip 传播、背书验证等协议栈行为与真实环境完全一致，因为使用的是同一套 Hyperledger Fabric 二进制代码
2. **网络行为真实**：BGP 路由收敛、AS 路径选择、OSPF 内部路由等网络行为基于真实协议栈（BIRD），非简单的延迟模拟
3. **可控变量实验**：精确控制网络参数（延迟/带宽/丢包）作为自变量，消除真实环境中的不可控因素
4. **成本效益**：单台 32 核 / 64GB 服务器即可运行 148 个容器的完整拓扑，无需多台物理机
5. **完全可复现**：所有配置脚本化，任何研究人员均可在相同硬件上复现相同实验结果
