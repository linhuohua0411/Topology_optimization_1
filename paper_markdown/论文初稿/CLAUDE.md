# CLAUDE.md — Topology Optimization 论文初稿写作规范（TIFS）

> 本文件用于约束 `paper_markdown/论文初稿/` 分章稿撰写与修订。  
> 统计口径与实验事实以 `results/statistics/`、`results/ablation_snapshots/`、`protocol/` 为唯一权威来源。  
> 任何定量结论必须可追溯到 `claim_traceability.md` 的 done claim。

---

## 一、核心原则

**所有文本围绕“网络演化动力学驱动的全局拓扑优化”主线展开，避免空泛网络优化叙事。**

必须严格执行：
1. 图表引用统一使用“**示出了**”；
2. 实验分析统一“**现象 -> 原因 -> 结论**”三段式；
3. 引言开篇采用“研究对象 -> 理想目标 -> 然而矛盾”结构；
4. 公式后统一用“其中...为...；...为...”解释变量；
5. 贡献句式统一“**针对...问题，提出了...**”；
6. 小节收束优先使用“**综上所述，...**”。

---

## 二、禁止用语（严格禁止）

| 禁止 | 替换为 |
|------|-------|
| “展示了/显示了/呈现了” | “示出了” |
| “效果很好/明显更优” | 给出数字或写“呈现提升趋势” |
| “我们认为/我们觉得” | 基于证据的客观陈述 |
| “显著提升”（无校正检验） | “有提升趋势/未达显著” |
| “主实验全部完成”（证据未闭环） | “当前主文定量闭环为 ED” |

---

## 三、各章节精细规范

### 【00-全文标题.md】

- 标题需体现：区块链网络 + 全局拓扑优化 + 演化动力学；
- 禁止标题先行承诺“最优/最强/通用 SOTA”。

---

### 【01-摘要.md】

- 一段式为主，建议 220-320 字；
- 顺序建议：问题背景 -> 现有不足 -> 方法核心 -> 实验设置 -> 关键结果 -> 安全意义；
- 只允许写入 `claim_traceability.md` 中 `status=done` 的定量主张；
- 禁止使用单次运行值或未校正 p 值作为“显著”依据。

---

### 【02-引言.md】

建议 5 段顺序：
- 段 1：区块链 P2P 拓扑鲁棒性问题及安全意义；
- 段 2：现有方法局限（静态性、预算公平性、统计口径不一致等）；
- 段 3：演化动力学与全局优化的必要性；
- 段 4：本文方案（“为此，本文提出...”）；
- 段 5：贡献点编号列表（与方法/实验章节一一对应）。

---

### 【03-相关工作.md】

- 结构建议：拓扑优化方法、鲁棒性评估方法、区块链网络实验实践；
- 每节需包含“代表工作 + 局限 + 本文切入点”；
- 不确定文献以 `[待核实]` 标注，不得作为核心论据。

---

### 【04-问题定义与威胁模型.md】

- 五字段完整：资产、对手能力、攻击面、目标、非目标；
- 与 `protocol/threat_model.md` 保持一致；
- 明确 ED-MAIN-BASE/LINK/NETEM 的攻击语义边界，避免跨场景混写。

---

### 【05-方法.md】

- 建议顺序：方法总览 -> 鲁棒性分解项（R/Rs/Rc/Rr）-> 全局优化策略 -> 预算约束 -> 复杂度；
- 明确 `gradient_mode=full` 为主文设置，`R_s_only/R_s_R_r` 仅用于消融；
- 禁止把消融结论前置为方法主贡献。

---

### 【06-实验设置.md】

- 必写项：数据来源、实验矩阵、基线全集、预算口径（fair/raw）、统计流程（Holm/FDR 分族）；
- 明确主文统计 CI 与检验口径与 `experiments/tifs_stats.py` 一致；
- 标注 `PROTOCOL_VERSION` 与关键脚本入口。

---

### 【07-主结果.md】

- 仅写 `unified_full_protocol` 主结果，默认以 fair 作为主排序口径；
- 定量证据来源：`stats_summary_graph_optimizer.{csv,json}` + 对应 run manifest；
- 若正文只展示部分基线，必须引用附录全基线表并声明比较全集。

---

### 【08-攻击类型分析.md】

- 按 BASE/LINK/NETEM 分节报告，不得跨攻击语义直接比较“谁更优”；
- 仅当有对应统计行时可写“显著”；
- 无统计闭环的条目改为“计划扩展”语气。

---

### 【09-消融与敏感性.md】

- C009（梯度组成）与 C004/C005（组件/MLE）分开写；
- 证据路径优先 `results/ablation_snapshots/`；
- 禁止把消融数字回填到 `07-主结果.md` 充当主结论。

---

### 【10-局限性.md】

- 局限必须与当前证据覆盖范围一致；
- 需明确：主文量化闭环重点在 ED，其他平台按当期证据谨慎表述；
- 不得在局限章节引入正文未出现的新结论。

---

### 【11-结论.md】

- 与 `01-摘要.md` 定量结论严格一致；
- 仅复述 done claim，不新增推断型数字；
- 结尾可给出后续计划，但应标注为未来工作而非既成事实。

---

### 【12-附录.md】

- 放置全基线表、补充统计表、参数细节、运行清单；
- 必须可支持正文所有关键结论回溯。

---

## 四、数字回填与一致性规则

1. 摘要、主结果、结论中同一指标必须同值同口径；  
2. “提升 x%”必须注明对照方法、场景、预算口径；  
3. “显著”只能基于修正后 p 值（Holm/FDR 对应分族）；  
4. 时间结论必须同时给出规模条件与预算条件；  
5. 禁止使用控制台临时输出直接入文。

## 六、质量自检清单

每次修改后逐项检查：

**内容层面**
- [ ] 每段仅有一个核心论点；
- [ ] 每个方法主张均有统计或协议证据；
- [ ] 实验段落满足“现象 -> 原因 -> 结论”；
- [ ] 局限性真实披露、无回避措辞。

**语言层面**
- [ ] 图表统一使用“示出了”；
- [ ] 公式变量解释完整；
- [ ] 术语首次出现有中英对应；
- [ ] 无夸大、无口语化判断。

**数字层面**
- [ ] 关键数字可回溯到 `results/` 与 manifest；
- [ ] 跨章节同指标一致；
- [ ] “显著”表述有修正后 p 值支持；
- [ ] 摘要与结论不出现 pending claim 的确定性语句。
